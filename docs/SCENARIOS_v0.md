# SCENARIOS_v0.md
## Сценарии работы системы испытаний (Surprize_v3.0)

Версия: v0  
Дата: 2026-02-13  
Назначение: Зафиксировать логику проведения испытаний, источники времени (t0), требования безопасности и обязательные артефакты.

---

# 0. Термины

- **Испытание (Trial)** — один полный цикл запуска сценария.
- **t0** — момент старта трансляции GPS-SDR-SIM, определяется по stdout-сообщению “transmission started”.
- **static_time** — время статики (сек), реализуется повторением первой точки траектории.
- **t_move_start** — момент начала движения по траектории:

  t_move_start = t0 + static_time

- **daemon-сервисы** — фоновые сервисы, работающие постоянно.
- **job-сервисы** — сервисы, участвующие в цикле выполнения (run-cycle).
- **Маяк (станок)** — подсистема управления дверью, шпинделями и моторами.
- **Шпиндели** — два шпинделя: основной (main) и стабилизирующий (stabilizing/head).

---

# 1. Глобальные условия безопасности (обязательные)

## 1.1 Контроль двери

Состояние двери определяется по положению двигателя двери (телеметрия Маяка).

Правила:

- Если дверь открыта:
  - Запуск испытания запрещён.
  - Ручной режим вращения разрешён, но с ограничением скорости и момента (см. 1.4).
- Если дверь открылась во время испытания:
  - Это аварийная ситуация (RED FLAG).
  - Испытание немедленно прерывается (ABORT).
  - Шпиндели останавливаются (команда на контролируемую остановку / аварийный профиль).
  - Видео и телеметрия продолжают записываться до полной остановки (best-effort).
  - В timeline.json фиксируется abort_reason.

---

## 1.2 Подтверждение закрепления устройства (v0)

Пока проверка закрепления реализована как подтверждение оператором.

- Подтверждение действует **только для одного испытания** (per-trial).
- Перед каждым новым испытанием оператор обязан подтвердить закрепление заново.
- После любого завершения испытания (FINISHED / ABORTED / ERROR) подтверждение сбрасывается.

---

## 1.3 Ручной режим (Manual Mode)

Ручное вращение шпинделей:

- Разрешено при открытой двери.
- Также разрешено при закрытой двери (с тем же безопасным ограничением, что и в испытании).
- Ограничено пределами RPM/момента (см. 1.4).
- Запуск испытания запрещён, пока активен ручной режим.
- При переходе в ARMED и RUNNING ручной режим должен быть отключён.
- Включение ручного режима во время RUNNING невозможно (запрещено на уровне UI и контроллера).

Если попытка активации ручного режима происходит во время RUNNING:
- Команда отклоняется.
- Логируется событие MANUAL_MODE_DENIED reason=trial_running.
- Состояние системы не меняется.

---

## 1.4 Ограничения по скорости и моменту (Safety Limits)

Для каждого шпинделя задаются лимиты, причём **лимиты разные**:
- для режима испытания (trial, дверь закрыта)
- для ручного режима при открытой двери (manual_door_open)

### Скорость (RPM)
- RPM ограничивается на уровне ввода/команд (в UI и в контроллере Маяка).
- Физически превысить RPM лимит шпиндель не сможет, но лимит остаётся контрактом команд.

### Момент (Torque) — software trip
- Ограничение момента — **мгновенное** (instant).
- Если actual_torque любого шпинделя превышает лимит для текущего режима → **немедленный ABORT**.

Аппаратная защита есть, но программная защита обязательна.

### Пример структуры настроек (концептуально)
majak:
  spindles:
    main:
      trial:           { max_rpm: ..., max_torque: ... }
      manual_door_open:{ max_rpm: ..., max_torque: ... }
    stabilizing:
      trial:           { max_rpm: ..., max_torque: ... }
      manual_door_open:{ max_rpm: ..., max_torque: ... }

# 2. Разделы интерфейса (UI)

## 2.1 Вкладка "Настройки"

Задаются редко изменяемые параметры:

### Инфраструктура / Видео
- Адрес Raspberry Pi (RTSP-стрим сервер):
  - ip / hostname
  - порт
- Параметры RTSP-клиента (режимы/таймауты)
- Параметры записи видео:
  - контейнер/кодек (например mkv)
  - поведение при потере сигнала (v0: placeholder-кадры включены)

### Ограничения безопасности (по шпинделям)
- Лимиты RPM/момента для каждого шпинделя:
  - trial (дверь закрыта)
  - manual_door_open (ручной режим при открытой двери)

### Временные параметры испытания
- spinup_lead_sec — за сколько секунд до t_move_start начать раскрутку
- параметры профиля торможения / деградации RPM
- прочие тайминги сценария вращения

### GPS-SDR-SIM / SDR
- параметры эфемерид
- частоты / прочие параметры генерации сигнала
- stdout marker text (если требуется настраивать)

### Пути
- trial_root — корневая папка хранения испытаний и артефактов

---

## 2.2 Планирование испытания

Оператор:

1) Генерирует траекторию (баллистика).
2) Задаёт static_time.
3) Задаёт сценарий вращения шпинделей:
   - Синхронная раскрутка.
   - Через заданный интервал стабилизирующий шпиндель начинает подтормаживать.
   - После достижения максимума RPM начинается убывающий профиль.

На этом этапе запись видео не ведётся.

---

## 2.3 Ручной режим

Используется для:

- Проверки механики.
- Предварительной настройки.

Ограничения:

- Скорость и момент ограничены лимитами текущего режима (см. 1.4).
- Испытание запускать нельзя, пока шпиндели вращаются вручную.

# 3. Состояния системы (циклическая модель)

Система работает циклически:

IDLE → READY → ARMED → RUNNING → (FINISHED | ABORTED | ERROR) → READY

## 3.1 Описание состояний

### IDLE
- Нет валидного плана,
  ИЛИ условия безопасности не выполнены (например дверь открыта и выбран сценарий испытания),
  ИЛИ испытание не может быть запущено по текущим условиям.

### READY
- План подготовлен.
- Дверь закрыта.
- Ручной режим выключен.
- Оператор может подтвердить закрепление.

### ARMED
- Подтверждение закрепления выполнено (per-trial).
- Создан trial_id.
- Зафиксирована конфигурация испытания (snapshot).
- Система готова к запуску.

### RUNNING
- Испытание выполняется.

### FINISHED
- Испытание завершено штатно.
- Все файлы закрыты.
- timeline.json записан.

### ABORTED
- Испытание остановлено аварийно.
- Причина зафиксирована.

### ERROR
- Инфраструктурная ошибка.
- Требуется вмешательство или восстановление.

---

После любого завершения (FINISHED/ABORTED/ERROR):

- Подтверждение закрепления сбрасывается (FixtureConfirmed := FALSE).
- Если условия безопасности соблюдены → переход в READY.
- Если нет → переход в IDLE.

# 4. Основной сценарий испытания (RUNNING)

## 4.1 Подготовка

1) Создаётся trial_id.  
2) Создаётся trial_dir.  
3) Стартует запись видео (2 канала) непрерывно.  
   - При потере потока пишутся placeholder-кадры (чёрные), чтобы шкала времени видео оставалась ровной.  
4) Стартует логирование телеметрии (примерно 1–10 Гц):
   - момент и скорость каждого шпинделя
   - угол стабилизирующего шпинделя  
5) Формируется trajectory_used (со статикой): первая точка повторяется так, чтобы обеспечить static_time.  
6) Запускается GPS-SDR-SIM с монолитной trajectory_used (static + dynamic в одном файле).  

## 4.2 Определение времени

- t0 фиксируется по stdout GPS-SDR-SIM (“transmission started”).
- t_move_start = t0 + static_time.

## 4.3 Фаза статики

- Шпиндели не вращаются.
- Видео и телеметрия записываются.

## 4.4 Фаза раскрутки

За spinup_lead_sec до t_move_start:

- Начинается синхронная раскрутка шпинделей.
- Через заданное время стабилизирующий шпиндель начинает подтормаживать (по профилю).

## 4.5 Основная фаза

- RPM изменяется по убывающему закону до указанного минимума.
- Телеметрия записывается.
- Видео записывается.
- При потере сигнала RTSP:
  - поток не останавливает испытание
  - запись продолжается placeholder-кадрами
  - интервалы потерь фиксируются в timeline.json

## 4.6 Контроль безопасности по моменту (в RUNNING)

- Момент каждого шпинделя контролируется по телеметрии (мгновенно).
- При превышении лимита момента для режима trial → немедленный ABORT (см. 5).

## 4.7 Завершение

- По окончании трансляции GPS (завершение SDR-процесса):
  - контролируемая остановка шпинделей
  - остановка телеметрии
  - завершение записи видео (finalize)
  - запись timeline.json
  - переход в FINISHED

---

# 5. Аварийный сценарий (ABORT)

Причины:

- Открытие двери во время RUNNING
- Команда оператора "Abort"
- Превышение допустимого момента (torque trip)
- Критическая ошибка инфраструктуры

Действия:

- Немедленная остановка вращения (аварийный/безопасный профиль)
- Видео и телеметрия продолжаются до фиксации завершения остановки (best-effort)
- В timeline.json фиксируется abort_reason
- Переход в ABORTED

  - static_time
  - параметры вращения

  - abort_reason (если есть)
  - пути к артефактам

# 7. Примечания / Follow-ups

- Демоны (видео, телеметрия) работают постоянно.
- Испытание (TrialRunner) будет job-сервисом (управляет сценарием).
- Точность синхронизации v0: 0.5–1 сек.
- stdout marker GPS-SDR-SIM должен быть зафиксирован точным шаблоном (regex) в настройках.
- Детальная схема CSV (telemetry/trajectory) будет закреплена в CONTRACTS.
